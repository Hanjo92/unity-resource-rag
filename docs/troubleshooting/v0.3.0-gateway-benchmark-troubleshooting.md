# v0.3.0 Gateway and Benchmark Troubleshooting

This guide covers the common setup and failure points for the `v0.3.0` gateway-first flow and the benchmark runners.

## Gateway Auth

Start the gateway with a token if you want the server to require bearer auth:

```bash
export UNITY_RESOURCE_RAG_GATEWAY_TOKEN=your-token
python3 -m pipeline.gateway
```

The server reports its supported adapters and capabilities at startup. For `v0.3.0`, the supported capabilities are:

- `vision_layout_extraction`
- `vision_layout_repair_analysis`
- `text_embedding`

If the gateway returns `401 auth_required`, check both sides:

- the gateway process has `UNITY_RESOURCE_RAG_GATEWAY_TOKEN` set
- the caller sends `Authorization: Bearer <token>`
- the workflow or planner is using the same token env name via `--gateway-auth-token-env`

If the gateway is reachable but the capability is rejected, verify the capability name matches one of the startup values and that the router supports the adapter you expect.

## Retrieval Benchmark Runner

Run the retrieval scorecard against a frozen suite manifest and a deterministic result payload:

```bash
python3 -m pipeline.evaluation.run_retrieval_benchmark \
  /absolute/path/to/benchmark-manifest.json \
  /absolute/path/to/retrieval-result.json
```

The input payload must match the suite manifest on `schemaVersion`, `benchmarkName`, and `projectName`. Each screen needs a `regions` array with `regionId`, `top1HitRate`, `top3HitRate`, and an optional candidate score field.

Common failures:

- `schemaVersion` mismatch
- missing screen or region results
- `selectedCandidateScore` below the fixture `minScore`
- low-confidence binding decisions such as `low_confidence_review`

## Screen Benchmark Runner

Run the screen scorecard against a benchmark report and the same suite manifest:

```bash
python3 -m pipeline.evaluation.run_screen_benchmark \
  /absolute/path/to/benchmark-manifest.json \
  /absolute/path/to/sample-benchmark-report.json
```

The report must match the suite manifest on `schemaVersion`, `benchmarkName`, and `projectName`. The runner compares each screen result against fixture thresholds and the expected mismatch classes in the report payload.

Common failures:

- missing or duplicate screen results
- top1/top3 hit rates below threshold
- `repairIterations` above the fixture limit
- unexpected issue types in the report

## Low-Confidence Binding

Binding policy matters when a candidate score is borderline:

- `best_match` always binds the top result
- `hold_if_uncertain` and `preserve_candidates` keep the query in the resolved blueprint and preserve alternatives in the report
- `require_confident` records a review-needed issue when the score is too low

If you want the workflow to continue without hard failure while still keeping the unresolved query visible, use `--allow-partial`.
