# v0.4.1-beta UX Follow-up Spec

Related backlog:

- [v0.4.1-beta UX Issue Backlog](./v0.4.1-beta-ux-issue-backlog.md)

## Release Theme

`v0.4.1-beta`의 목표는 `unity-resource-rag`를 "Unity 실프로젝트에서 end-to-end가 되는 beta"에서 "처음 쓰는 Unity 사용자도 창 하나 안에서 첫 성공까지 갈 수 있는 beta"로 끌어올리는 것이다.

한 줄로 정리하면:

`Quick Setup -> Start UI Build -> Capture & Repair`가 Unity 창 안에서 끊기지 않고, 사용자는 MCP transport, resource/tool 차이, Python runtime 같은 내부 개념을 거의 몰라도 된다.

## Context And Assumptions

### Confirmed

- `0.4.0-beta` 기준으로 real Unity project에서 `catalog -> blueprint -> Unity output` 플로우는 검증됐다.
- `Window > Unity Resource RAG`에 `Quick Setup`과 `Start UI Build` entrypoint가 들어왔다.
- `Start UI Build`는 현재 background runner로 돌아가며, real project에서 오브젝트 생성까지 성공했다.
- provider/auth 기본 경로는 `Use existing Codex login`이 가장 마찰이 적다.

### Observed Friction

- 사용자는 여전히 local checkout, Python interpreter, MCP HTTP Local, custom tool/resource 같은 내부 개념을 알아야 할 때가 있다.
- `doctor`와 build report에는 아직 MCP 내부 용어가 남아 있어 사용자 언어로 읽기 어렵다.
- build 이후 screenshot capture와 repair loop는 아직 Unity 창 안의 단일 happy path로 닫혀 있지 않다.
- `catalog draft`는 popup 중심이라 HUD/list/inventory 계열로 일반화가 약하다.
- 실프로젝트 품질 보강을 위해 case capture는 필요하지만, 현재는 수동 문서화 비중이 높다.

### Assumed

- 기준 Unity stack은 계속 `2021.3+`, `UGUI`, `TextMeshPro`다.
- `unity-mcp`는 계속 transport/runtime host를 담당하고, `unity-resource-rag`는 그 위에 얹는 애드온처럼 동작한다.
- `v0.4.1-beta`에서는 broad provider expansion보다 Unity-side usability를 우선한다.

## Why This Release

`0.4.0-beta`는 "실제로 된다"를 증명한 릴리즈였다.

다음 단계에서 필요한 것은 capability 추가보다 아래 네 가지다.

1. 사용자가 terminal과 MCP 설정을 몰라도 첫 성공까지 갈 수 있게 만들기
2. setup/build/capture/repair를 Unity 창 기준으로 한 경로에 모으기
3. 내부 진단 언어를 사용자 언어로 바꾸기
4. 실제 프로젝트 케이스를 같은 형식으로 수집하게 만들기

## Product Goals

### Goal 1. First-Click Readiness

Unity 창이 "지금 무엇이 준비됐고 무엇이 막히는지"를 MCP jargon 없이 알려줘야 한다.

### Goal 2. Single-Window Happy Path

사용자는 `Quick Setup`, `Start UI Build`, `Capture & Repair` 정도의 흐름만 알면 된다.

### Goal 3. User-Facing Language

보고서와 경고는 `resource missing`, `project-scoped tools`, `custom tool exposure` 같은 내부 용어보다 사용자가 행동으로 옮길 수 있는 문장으로 보여야 한다.

### Goal 4. Reusable Quality Cases

실프로젝트 생성 결과를 screenshot/artifact 중심으로 저장하고, 다음 품질 개선의 backlog 입력으로 재사용할 수 있어야 한다.

## Non-Goals

- 새로운 provider vendor 대량 추가
- UI Toolkit 지원
- multi-project generalized autopilot
- gateway 분산 구조 확장
- full auto-healing loop

## v0.4.1-beta Success Criteria

릴리즈 완료 기준은 아래를 모두 만족하는 것이다.

1. 처음 설치한 사용자가 Unity 창에서 readiness 상태를 보고, terminal 없이 첫 build를 시작할 수 있다.
2. `Quick Setup`이 transport/tool visibility/runtime/auth readiness를 같은 보고서 안에서 설명한다.
3. `Start UI Build`는 blocking 없이 phase progress를 보여준다.
4. build 성공 뒤 같은 창에서 screenshot capture 또는 repair 진입이 가능하다.
5. 사용자-facing 보고서가 내부 MCP resource/tool jargon 없이도 이해 가능하다.
6. 실프로젝트 case를 artifact bundle + screenshot + notes 형태로 export 또는 issue-ready report로 남길 수 있다.

## Release Gates

### Gate A. Readiness UX

- readiness card가 `repo`, `python`, `codex auth`, `unity mcp`, `build inputs`를 분리해서 보여준다.
- 각 상태는 `ready`, `attention`, `blocked`로 요약된다.

### Gate B. Happy Path UX

- `Start UI Build`는 실행 중에도 Unity editor를 멈추지 않는다.
- 성공 시 `blueprint`, `Unity apply`, `next step`을 사용자 언어로 보여준다.

### Gate C. In-Window Follow-Up

- 최소 screenshot capture 진입이 Unity 창 안에서 가능하다.
- repair는 최소 handoff 생성까지 같은 창에서 연결된다.

### Gate D. Quality Evidence

- 실프로젝트 case 3건 이상이 같은 형식으로 수집된다.
- popup, HUD, list/inventory 중 최소 2개 패턴이 포함된다.

## Architecture Decisions

### 1. Unity Window를 user-facing product surface로 본다

CLI/MCP tool 자체는 계속 유지하되, `v0.4.1-beta`의 primary UX는 Unity window 기준으로 설계한다.

### 2. Internal diagnostics와 user-facing summaries를 분리한다

`doctor`와 raw payload는 내부 디버깅용으로 유지하고, Unity window에는 condensed summary를 별도 formatter로 제공한다.

### 3. Runtime bootstrap은 settings보다 readiness/repair loop를 우선한다

Python/runtime/auth 감지는 settings asset에 저장하되, 사용자 경험은 "설정값 편집"이 아니라 "지금 바로 고칠 수 있는 readiness action" 중심으로 둔다.

### 4. Follow-up flow는 build 이후 screenshot/repair까지 닫는다

`Start UI Build`가 끝나면 사용자가 다음에 해야 할 행동을 다시 추론하지 않게 한다.

### 5. Quality learning은 docs-only가 아니라 artifact-first로 남긴다

케이스 수집은 markdown memo보다 screenshot path, blueprint path, generated object name, notes가 포함된 structured report를 우선한다.

## Epic Breakdown

### Epic 1. Readiness And Setup UX

목표:

- setup readiness를 사용자 언어로 보여주고, 가능한 것은 창 안에서 바로 고친다.

대표 범위:

- readiness card
- Python/runtime/bootstrap guidance
- auth preset simplification
- MCP jargon 제거

### Epic 2. Build Execution UX

목표:

- build를 non-blocking으로 돌리고 progress와 결과를 읽기 쉽게 만든다.

대표 범위:

- background execution
- phase progress
- better result summary
- template mode selection

### Epic 3. Follow-Up Loop UX

목표:

- screenshot capture와 repair handoff를 같은 창에서 연결한다.

대표 범위:

- capture button
- last build artifact memory
- repair entrypoint

### Epic 4. Quality Case Capture

목표:

- 실프로젝트 사례를 같은 형식으로 모으고 이후 retrieval/repair backlog 입력으로 재사용한다.

대표 범위:

- case export
- issue template wiring
- artifact bundle summary

주의점:

- Unity window는 raw MCP payload를 그대로 노출하는 디버그 콘솔이 되지 않게 한다.
- provider/auth 결정 로직은 Python sidecar 쪽에서 유지하고, Unity UI는 selection과 readiness 표현만 담당한다.
- bootstrap automation은 가능한 범위만 자동화하고, destructive install/update는 명시적 사용자 액션으로 둔다.

## Suggested Release Positioning

`v0.4.1-beta`는 capability release가 아니라 usability release다.

메시지는 아래가 적합하다.

- real-project validation 이후, Unity-first beta UX를 다듬는 릴리즈
- terminal/MCP 지식 없이 첫 build success에 도달하는 마찰을 줄이는 릴리즈
- screenshot/repair와 quality case capture까지 잇는 operator-friendly beta
