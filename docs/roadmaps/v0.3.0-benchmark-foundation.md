# v0.3.0 Benchmark Foundation

This document defines the minimum benchmark scaffold for `v0.3.0`.

Goal:

- freeze benchmark inputs early
- make fixture loading explicit
- give runners and gate publishing a stable file layout

## Scope

The scaffold now covers five things.

1. Suite manifest
2. Retrieval fixtures and screen fixtures
3. Run report shape
4. Retrieval and screen benchmark runners
5. Baseline gate artifact publishing

## File Layout

```text
examples/benchmarks/
  README.md
  v0.3.0-reference-suite/
    benchmark-manifest.json
    benchmark-thresholds.json
    sample-retrieval-result.json
    sample-benchmark-report.json
    baseline-artifacts/
      retrieval-scorecard.json
      screen-scorecard.json
      benchmark-gate-report.json
      benchmark-gate-report.md
    reward-popup/
      retrieval-fixture.json
      screen-fixture.json
    inventory-panel/
      retrieval-fixture.json
      screen-fixture.json
pipeline/evaluation/
  __init__.py
  fixtures.py
  models.py
  publish_benchmark_gate_report.py
  report_models.py
  run_retrieval_benchmark.py
  run_screen_benchmark.py
```

## Acceptance Criteria

- the suite manifest loads without custom parsing glue
- each screen fixture resolves to a retrieval fixture, screen fixture, and threshold file
- the report model can load the sample benchmark report
- retrieval and screen scorecards are reproducible from checked-in sample inputs
- a portable benchmark gate report can be published without machine-local absolute paths
- schema validation fails fast on missing keys

## Notes

- Example image paths are placeholders and are meant to be replaced with real benchmark assets.
- The fixture schema is intentionally small so benchmark runners can evolve without rewriting the loading layer.
- The checked-in gate report is still fixture-backed; final release sign-off should rerun it against real captures.
