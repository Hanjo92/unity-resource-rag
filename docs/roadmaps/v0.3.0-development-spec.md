# v0.3.0 Development Spec

Related backlog:

- [v0.3.0 Issue Backlog](./v0.3.0-issue-backlog.md)

## Release Theme

`v0.3.0`의 목표는 `unity-resource-rag`를 "혼자 로컬에서 한 번 돌려보는 도구"에서 "팀이 gateway-first 경로로 반복 검증할 수 있는 UI reconstruction pipeline"으로 끌어올리는 것이다.

한 줄로 정리하면:

`reference image -> gateway-backed extraction/repair analysis -> benchmarked retrieval -> Unity apply -> screenshot verify -> bounded repair patch`

## Context And Assumptions

### Confirmed

- `v0.2.1` 기준으로 MCP/CLI provider preset 정규화와 auth inspection 회귀는 정리되어 있다.
- `pipeline.gateway`는 최소 vertical slice로 들어와 있고, 현재는 `vision_layout_extraction` + `gemini_direct`만 지원한다.
- retrieval은 여전히 로컬 `tfidf-sparse-v1` 하이브리드 검색이 기본이다.
- verification은 mismatch 분석과 repair handoff 생성까지 가능하지만, patch semantics와 benchmark 기준은 아직 약하다.
- Unity 실행 경계는 계속 `unity-mcp` + UPM package가 담당한다.

### Assumed

- 기준 Unity stack은 계속 `2021.3+`, `UGUI`, `TextMeshPro`다.
- `v0.3.0`에서도 실제 benchmark는 Unity 프로젝트 1개를 기준으로 진행한다.
- 팀 공유 환경에서는 provider key를 sidecar/MCP client보다 gateway 쪽에 두는 방향을 계속 우선한다.

### Proposed

- `v0.3.0`에서는 기준 화면을 1개에서 2개로 늘린다.
- 단, 서로 완전히 다른 장르의 화면으로 넓히지 않고, 같은 프로젝트 안에서 구조가 다른 두 화면만 본다.
- 추천 조합은 `canonical popup` 1개 + `list/card or slot-based panel` 1개다.

## Why This Release

`v0.1.0`이 "뼈대와 계약이 있는 상태"였다면, `v0.2.x`는 "한 화면을 끝까지 닫을 수 있는 흐름을 정리한 상태"에 가깝다.

다음 단계에서 필요한 것은 breadth가 아니라 아래 세 가지다.

1. `gateway`를 문서상 옵션이 아니라 실제 팀 기본 경로로 만들기
2. retrieval / verification / repair를 감이 아니라 benchmark로 다루기
3. 한 화면 특화가 아니라 "작게 일반화된" 두 번째 화면까지 닫아보기

## Product Goals

### Goal 1. Gateway-First Default

공유 환경에서 `--provider gateway` 또는 `auto -> gateway`가 실제 권장 기본 경로가 되게 한다.

### Goal 2. Measurable Quality

retrieval과 screenshot verification이 benchmark fixture와 scorecard로 비교 가능해야 한다.

### Goal 3. Bounded Repair

repair 결과가 설명문이 아니라, node/anchor/layout/asset 단위의 구조화된 patch candidate로 내려와야 한다.

### Goal 4. Small Generalization

기준 화면 2개를 통과시켜 "특정 popup 하나에 과적합된 흐름"에서 벗어난다.

## Non-Goals

- provider 종류를 대폭 늘리는 것
- gateway를 분산 시스템이나 다중 서비스 구조로 확장하는 것
- UI Toolkit 지원 추가
- 완전 자동 다회차 self-healing loop
- 외부 vector DB 의존 도입
- 대규모 multi-project generalization

## v0.3.0 Success Criteria

릴리즈 완료 기준은 아래를 모두 만족하는 것이다.

1. `gateway`가 최소 두 capability를 안정적으로 제공한다.
   - must-have: `vision_layout_extraction`
   - must-have: `vision_layout_repair_analysis`
   - should-have: `text_embedding`
   - optional stretch: `image_embedding`
2. retrieval benchmark fixture가 repo 안에 정의되어 있고, 반복 실행 가능한 runner가 존재한다.
3. 기준 화면 2개에 대해 region-level retrieval scorecard를 생성할 수 있다.
4. 기준 화면 2개에 대해 end-to-end artifact bundle과 verification report를 같은 절차로 재현할 수 있다.
5. repair output이 patch candidate schema를 통해 `asset`, `layout`, `anchor`, `spacing`, `component` 수준으로 분류된다.
6. 실패 원인이 최소 아래 taxonomy로 구분된다.
   - `provider_auth`
   - `provider_runtime`
   - `catalog_quality`
   - `retrieval_low_confidence`
   - `binding_failure`
   - `unity_apply_failure`
   - `layout_mismatch`
   - `asset_style_mismatch`
   - `repair_no_confident_patch`

## Release Gates

### Gate A. Benchmark Baseline

- 기준 Unity 프로젝트 1개와 benchmark 화면 2개가 고정된다.
- 각 화면마다 reference image, expected region fixture, 성공 screenshot 예시가 있다.

### Gate B. Gateway Capability Expansion

- 동일한 gateway boundary에서 extraction과 repair analysis를 호출할 수 있다.
- auth failure와 capability failure가 normalized error contract로 분리된다.
- sidecar artifact에는 provider-neutral metadata만 남고 secret은 남지 않는다.

### Gate C. Retrieval Quality

- fixture 기준 핵심 region top-1 hit rate가 `>= 0.70`
- fixture 기준 핵심 region top-3 hit rate가 `>= 0.90`
- low-confidence case는 억지 auto-bind 대신 candidate 보존 경로로 떨어진다.

### Gate D. Screen-Level Output Quality

- 기준 화면 2개 모두 `workflow-report`와 `verification-report`를 생성한다.
- 각 화면은 최대 1회 repair iteration 안에 아래 둘 중 하나를 만족한다.
  - `hasMeaningfulMismatch = false`
  - 또는 `normalizedMeanAbsoluteError <= 0.10` 이고 `foregroundBboxIoU >= 0.75`

### Gate E. Reproducibility

- setup guide, benchmark run guide, troubleshooting guide가 존재한다.
- 다른 팀원이 같은 입력으로 같은 benchmark runner를 실행할 수 있다.

## Architecture Decisions

### 1. `gateway`는 team-safe default로 진화시킨다

직접 provider key를 sidecar에 뿌리는 경로는 개발 편의용으로 유지하되, 팀 기본 경로는 `gateway`를 기준으로 잡는다.

### 2. Provider surface가 아니라 capability surface를 늘린다

`v0.3.0`의 확장 단위는 vendor가 아니라 capability다.

우선순위:

- `vision_layout_extraction`
- `vision_layout_repair_analysis`
- `text_embedding`
- `image_embedding`은 stretch

### 3. Retrieval baseline은 제거하지 않는다

로컬 `tfidf-sparse-v1`은 fallback baseline으로 유지한다. `v0.3.0`의 목표는 baseline 교체가 아니라 benchmark 가능한 rerank layer 추가다.

### 4. Repair는 prompt가 아니라 patch candidate를 산출한다

repair 결과는 "어디가 이상하다"에서 멈추지 않고, bounded edit proposal을 내려야 한다.

예:

- `replace_asset`
- `swap_prefab_candidate`
- `adjust_anchor`
- `adjust_size_delta`
- `adjust_spacing`
- `review_component_contract`

### 5. Repo에는 fixture와 report를 남기고, 실프로젝트 자산은 남기지 않는다

benchmark 재현에 필요한 것은 JSON fixture, screenshot sample, score report다. 실제 사내 프로젝트 asset 자체를 repo에 넣는 방향은 피한다.

## Bounded Contexts And Responsibilities

### 1. Sidecar Orchestration

책임:

- workflow entrypoint
- artifact generation
- provider-neutral report 생성
- Unity MCP handoff 생성

### 2. Gateway Capability Layer

책임:

- capability routing
- auth normalization
- upstream adapter 선택
- normalized usage/trace/error 반환

### 3. Retrieval Evaluation Layer

책임:

- fixture 정의
- region-level ranking benchmark
- score aggregation
- regression diff report

### 4. Verification And Repair Layer

책임:

- screenshot mismatch analysis
- suspect node ranking
- patch candidate 제안
- repair report scoring

## Proposed Repository Changes

아래 정도의 구조 보강을 권장한다.

```text
pipeline/
  evaluation/
    fixtures.py
    report_models.py
    run_retrieval_benchmark.py
    run_screen_benchmark.py
  gateway/
    adapters/
    capabilities/
      vision_layout_extraction.py
      vision_layout_repair_analysis.py
      text_embedding.py
  verification/
    build_repair_patch_candidates.py
examples/
  benchmarks/
    <benchmark-name>/
      retrieval-fixture.json
      screen-fixture.json
      expected-thresholds.json
docs/
  benchmark-run-guide.md
  troubleshooting/
    gateway-auth.md
    retrieval-diagnostics.md
```

주의점:

- `pipeline/gateway/server.py` 하나에 capability branching을 계속 밀어넣지 않는다.
- benchmark runner는 workflow script를 재활용하되, artifact 채점 규칙은 `pipeline/evaluation`로 분리한다.
- repair patch builder는 기존 `build_repair_handoff_bundle.py`와 책임이 겹치지 않게 `report -> patch candidates` 경계로 둔다.

## Epic Breakdown

### Epic 1. Benchmark Project And Fixture Definition

목표:

- `v0.3.0`의 성공 여부를 같은 기준으로 측정할 benchmark 입력을 고정한다.

산출물:

- benchmark project memo
- target screens 2개 정의
- `retrieval-fixture.json`
- `screen-fixture.json`
- expected threshold document

완료 기준:

- 팀이 같은 화면과 같은 scoring 기준으로 얘기할 수 있다.

### Epic 2. Gateway Capability Expansion

목표:

- `gateway`가 extraction-only slice에서 repair/embedding-aware boundary로 확장된다.

산출물:

- capability router 구조
- `vision_layout_repair_analysis` contract
- `text_embedding` contract
- normalized capability error examples

완료 기준:

- planner / verification / evaluation이 같은 gateway 경계로 호출 가능하다.

### Epic 3. Retrieval Benchmark And Rerank V2

목표:

- retrieval 품질을 benchmark로 올리고, low-confidence path를 안전하게 만든다.

산출물:

- retrieval benchmark runner
- before/after score report
- candidate preservation rule
- preview-aware rerank 설계 또는 구현

완료 기준:

- top-1 / top-3 개선 여부가 숫자로 보인다.

### Epic 4. Structured Repair Patch Pipeline

목표:

- verification 결과를 사람이 이해하기 쉬운 patch candidate 집합으로 바꾼다.

산출물:

- repair patch schema
- issue-to-patch mapping rules
- patch candidate ranking
- repair handoff integration note

완료 기준:

- repair 결과가 단순 prompt가 아니라 bounded action set이 된다.

### Epic 5. Reproducible Benchmark Flow

목표:

- 다른 팀원이 같은 입력으로 같은 benchmark를 돌릴 수 있게 한다.

산출물:

- benchmark runner CLI
- benchmark run guide
- setup / auth / troubleshooting 문서
- sample report artifact

완료 기준:

- 다른 사람이 repo 문서만 보고 benchmark 실행까지 도달 가능하다.

## Risks

### Risk 1. Gateway capability expansion이 실제 품질 개선 없이 구조 작업으로만 끝날 수 있음

대응:

- capability 추가는 반드시 benchmark runner 연결과 함께 진행한다.

### Risk 2. Retrieval 개선보다 catalog 품질 부족이 더 큰 병목일 수 있음

대응:

- fixture와 low-confidence taxonomy를 먼저 만들고, retrieval blame을 분리한다.

### Risk 3. Repair patch가 너무 공격적이면 잘못된 수정 자동화를 유도할 수 있음

대응:

- `v0.3.0`에서는 patch candidate 생성까지만 하고 auto-apply는 목표에서 제외한다.

### Risk 4. 화면 2개로 늘리면서 scope가 갑자기 커질 수 있음

대응:

- 같은 프로젝트 안의 구조적으로 가까운 두 화면만 선정한다.

### Risk 5. `image_embedding` 범위가 커서 전체 일정이 흔들릴 수 있음

대응:

- text embedding과 benchmark runner를 must-have로 잡고, image embedding은 stretch로 둔다.

## Open Decisions

1. benchmark project를 공개 가능한 샘플 프로젝트로 둘지, 내부 프로젝트로 둘지
2. `image_embedding`을 `0.3.0`에 포함할지 `0.3.x`로 미룰지
3. repair patch artifact를 Unity MCP가 직접 소비하게 할지, 사람이 review 후 handoff로 변환하게 할지

## What v0.3.0 Should Feel Like

`v0.3.0`은 "`gateway`가 실제 기본 경로로 작동하고, 두 개의 benchmark 화면에 대해 품질과 실패 원인을 숫자와 artifact로 설명할 수 있는 상태"여야 한다.
