# 0.6.1

`unity-resource-rag` 0.6.1 is a patch release focused on trimming end-to-end wait time around `Start UI Build`, catalog-first draft generation, and Unity-side blueprint application.

## Highlights

- reduced repeated Unity asset resolution and reflection work during blueprint validation and apply
- reused doctor-discovered Unity MCP tool availability during build startup instead of querying the same inventory again
- removed repeated subprocess churn from catalog-first retrieval and the reference-to-resolved-blueprint workflow
- keeps the post-`0.6.0` Unity MCP menu bridge and readiness-flow improvements in the released package

## Included Changes

### Unity apply path

- added caches for resolved asset references, sub-assets, primary asset loads, and runtime type resolution in `UnityAssetResolver`
- cached reflection lookups in `ApplyUiBlueprintTool` so repeated `Type`, `PropertyInfo`, and `FieldInfo` discovery does not scale linearly with large blueprints

### Build orchestration

- `start_ui_build` now runs doctor in a build-focused scope by default so build preflight skips `resources/list` when it is not needed
- doctor-discovered Unity tool names are forwarded into first-pass and catalog-draft routes as `available_tools_hint`
- catalog-first retrieval reuses loaded catalog records and vector-index payloads instead of spawning repeated search subprocesses
- the reference image workflow now runs local blueprint/template, vector-index, binding, and handoff steps in process where possible

## Validation

- `python3.12 -m unittest tests.test_doctor_tool tests.test_start_ui_build_tool tests.test_catalog_draft_tool tests.test_first_pass_tool tests.test_bind_blueprint_assets`
- `python3.12 -m compileall pipeline`

## Notes

- the long-lived untracked example draft blueprint remains outside the release payload
- reference-image model inference time is still the largest remaining latency source when a build includes vision extraction
