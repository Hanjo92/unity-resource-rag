# 0.2.1

Provider runtime and release polish for `unity-resource-rag`.

## Highlights

- Added a minimal provider gateway module that exposes `POST /v1/capabilities/run`.
- Added the first gateway adapter vertical slice for Gemini-based `vision_layout_extraction`.
- Tightened provider setup inspection so custom OpenAI-compatible endpoints require explicit auth instead of inheriting Codex OAuth.
- Fixed `connection_preset` normalization so stale auth fields no longer override the selected preset.
- Added regression tests for provider setup inspection and preset normalization.

## Included In This Release

### Provider Runtime

- `gateway` provider path in planner inspection and workflow runtime
- gateway URL and bearer token support in the reference layout planner
- normalized provider setup guidance for `openai`, `openai_compatible`, and `gateway`

### Gateway

- `pipeline.gateway` HTTP server
- Gemini direct adapter
- gateway auth helpers and normalized response models
- local bootstrap script for Gemini OAuth token files

### Docs And Contracts

- provider gateway architecture overview
- provider auth matrix
- provider gateway contract
- updated MCP client setup and workflow README guidance

### Quality

- provider setup regression tests
- local startup and auth smoke verification for the MCP server

## Notes

- `openai_compatible` now defaults to explicit provider credentials. Codex OAuth auto-reuse remains supported for `openai`, but not for custom-compatible endpoints.
- `connection_preset` now clears conflicting auth fields before runtime inspection so the preset remains authoritative.

## Recommended Next Step

Use `0.2.1` to validate both auth paths in a real project:

1. direct `openai` via Codex OAuth or API key
2. `openai_compatible` via provider-specific API key or explicit OAuth token input
3. `gateway` via local or team-hosted gateway
