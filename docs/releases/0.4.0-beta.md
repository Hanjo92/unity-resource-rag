# 0.4.0-beta

Real-project validation release for `unity-resource-rag`.

## Release Positioning

This release marks the first verified real Unity project run where `unity-resource-rag` completed the end-to-end flow from project resource catalog generation to blueprint creation and Unity-side asset/application.

This is a beta release focused on functional validation rather than broad quality sign-off. Output quality can still vary based on project asset density, naming consistency, reference quality, and UI reuse structure.

## Highlights

- Verified `index_project_resources` in a real Unity project instead of repo-only or synthetic validation.
- Verified the project catalog could be used to generate a UI blueprint and drive Unity-side output creation.
- Confirmed the Unity integration path works with `unity-mcp` in HTTP mode with project-scoped custom tool discovery disabled.
- Closed the loop on the package's core MVP promise: `catalog -> blueprint -> Unity output`.

## Included In This Release

### Real Project Validation

- real Unity project package installation
- Unity compile fixes for custom tool assembly references
- Unity-side custom tool discovery validation for `index_project_resources` and `apply_ui_blueprint`
- successful resource catalog generation in a non-demo project
- successful blueprint-driven Unity output generation from that catalog

### Stability Improvements

- editor assembly reference fix so `ToolParameterAttribute` resolves correctly against `unity-mcp`
- definite-assignment fix in `ApplyUiBlueprintTool`
- clarified setup/debug path for Unity MCP HTTP local mode and custom tool visibility

## Known Limitations

- output quality is still highly dependent on project asset coverage and asset naming quality
- sparse or inconsistent UI asset libraries can reduce retrieval quality and blueprint fidelity
- some screens may still require manual cleanup or repair-loop iteration after first-pass generation
- current validation evidence is still based on a limited number of real project cases

## Recommended Next Step

- ship this as a beta milestone
- collect 3 to 5 real-project cases across popup, HUD, and list-style UIs
- use those cases to improve retrieval ranking, blueprint fidelity, and repair guidance before the next stability-oriented release
